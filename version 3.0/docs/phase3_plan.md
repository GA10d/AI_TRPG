**需求冻结版**

我按你刚刚的修正，整理成这一版开发规格。后面我会按这个规格进实现，不先写代码之外的方案漂移。

**1. 游戏主界面**
1. 主界面按“核心游玩操作优先”设计，不再是信息堆砌页。
2. 一级可见内容固定为：`最新一轮 AI 叙事`、`玩家输入框`、`最近上下文/历史对话`、`保存`、`读档`、`NPC`、`详情`。
3. `保存/读档` 保留在主界面一级入口，不收进详情。
4. `NPC` 作为一级入口，打开独立 NPC 界面，用于立绘展示和人物信息查看。
5. `详情` 负责承接非核心信息：调试信息、结构化结果原文、模型信息、参数、回放日志、分支图等。
6. 页面视觉上会以“叙事阅读 + 输入行动”为中心，最新 AI 输出是视觉主焦点，历史对话次级展示，操作区稳定停留在用户手边。
7. 我会按职业项目经理视角做成“能长期游玩”的主界面，而不是只满足技术演示。

**2. Agent 方案**
1. MVP 先做两个 Agent：
2. `Narrator Agent`：负责玩家可见叙事。
3. `Ending Judge Agent`：只判断这一轮后是否进入结局。
4. `Ending Judge Agent` 每轮只吃 `Narrator Agent 本轮输出`，不吃完整上下文。
5. 它的结构化输出先压到最小：核心只有 `是否进入结局`，必要时再附 `endingState`。
6. 我会在 `version 3.0/apps/prompt/single_agent/` 里新增这个判定 Agent 的 system prompt，默认文件名我会用 `ending_judge_prompt.txt`。

**3. Narrator Prompt 与开局调用**
1. 讲故事 Agent 的 system prompt 以 `version 3.0/apps/prompt/single_agent/narrator_prompt.txt` 为准。
2. 开局时，不再把用户文本拼成一个新的 `player_info.txt`。
3. 正确理解是：用户在 `beginning/角色创建阶段` 写的那段文本，本身就是 player info。
4. 正式开始游戏时，第一次发给 Narrator Agent 的输入内容是：`rule.txt`、`story.txt`、`用户填写的 player info 文本`。
5. 也就是说：`system prompt` 和 `首轮用户输入材料` 分离，前者走 `narrator_prompt.txt`，后者走这三份内容。

**4. 文生图 Pipeline**
1. `3.0` 会补一套文生图接口底座，参考 `2.0` 的实现方式先搭通。
2. Prompt 生成方式采用：`业务 prompt + 通用模板 prompt` 拼接。
3. 通用模板按场景类型拆分，至少预留：`character_portrait`、`npc_intro`、`scene_shift`、`manual`。
4. 这些通用模板需要暴露出来供你配置，而不是写死在代码里。
5. 第一阶段先把接口、配置结构、调用链和占位 UI 建好，后续再往 NPC 立绘和场景图继续扩。

**5. 设置页**
1. 设置页除了文本模型配置，还要新增 image provider 配置。
2. 需要覆盖：provider、api key、base url、model，以及通用模板 prompt 的配置入口。
3. 这套配置会和文本模型配置并列，而不是混在一个字段里。

**6. 这一轮范围里不主打的内容**
1. 结构化 Agent 这轮不做复杂状态同步，只做结局判断。
2. 主界面不再默认摊开所有参数、日志和调试数据。
3. 文生图这轮先搭底座，不追求一次把所有图像玩法做完。
